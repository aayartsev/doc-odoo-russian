      Доступно на GitHub

      Помимо ручного управления доступом с помощью кода, Odoo предоставляет два основных механизма на основании данных в модуле для управления или ограничения доступа к данным в базе.

      Оба механизма связаны с нужными пользователями через группы: пользователь принадлежит какому-то количеству групп, и механизмы безопасности ассоциированы с группами, таким образом эти механизмы применяются к пользователям.

      Контроль доступа

      Управляемый записямиir.model.access, определяет доступ ко всей модели данных.

      Каждый элемент управления доступом имеет модель данных, к которой предоставляет разрешения, разрешения он предоставляет и опционально группу.

      Контроль доступа является аддитивными, для данной модели пользователь имеет доступ ко всем разрешениям, предоставленным любой из его групп: если пользователь принадлежит к одной группе, которая позволяет писать и к другой, которая позволяет удалять, он может и писать и удалять.

      Если не указано ни одной группы, контроль доступа применяется ко всем пользователям, в противном случае это относится только к членам данной группы.

      Доступные разрешения это создание (perm_create), поиск и чтение (perm_read), обновление существующих записей (perm_write) и удаление существующих записей (perm_unlink)

      Правила записей

      Запись правил это условие, что записи удовлетворяют разрешенным операциям (создание, чтение, обновление или удаление). Она применяется от записи к записи после того, как будет применен контроль доступа.

      Запись правила имеет:

      • модель данных, к которой она применяется
      • набор разрешений, к которому он относится (например, если perm_read установлен, то правило будет проверяться только при чтении записи)
      • набор групп пользователей, к которым применяется данное правило, если ни одна из групп не указа, тогда правило становится global
      • domain используется для проверки, соотвествует ли данная запись правилу (и доступна) или не соотвествует (и не доступна). Домен оценивается с двумя переменными в контексте: user принадлежит ли эта запись пользователю и time проверяется доступность по времени

      Глобальные правила и правила для конкретной группы (правила ограничиваются конкретными группами против групп, принадлежащих всем пользователям) используются совершенно по-разному:

      • Глобальные правила являются вычитаемыми, они должны все соотвествовать записи, чтобы получить доступ
      • Правила для групп являются аддитивными, если хотябы одно из них совпадает (и все глобальные правила совпадают) тогда запись доступна

      Это означает, что первое правило группы ограничивает доступ, но любое другое правило группы расширяет его, в то время как глобальные правила могут только ограничивать доступ в дальнейшем (или не иметь эффекта).

      Внимание

      Правила записей не применяются к пользователю Administrator

      в отличии от правил доступа

      Доступ к полям

      Доступно с версии 7.0.

      ORM поле может иметь атрибут группы предоставляющий список групп (в виде разделенных запятыми строки внешних идентификаторов).

      Если текущий пользователь не находится в одной из перечисленных групп, он не будет иметь доступ к полю:

      • ограниченные поля автоматически удаляются из запрошенных представлений
      • ограниченные поля удаляются из ответов на fields_get() запросы
      • попытки (в явном виде) прочитат или записать в ограниченные поля приводит к ошибке доступа

      Workflow transition rules

      Переходы между этапами бизнес процесса могут быть ограничены до нескольких групп пользователей. Пользователи вне этих групп не смогут совершить перевод процесса на следующий этап.
