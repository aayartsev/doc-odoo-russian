      Маршрутизация

      openerp.http.route(route=None, **kw)[source]

      Это декоратор маркирующий декорируемый метод как обработчик запросов. Метод должен быть частью подкласса Controller.

      Параметры
      • route -- строка или массив. Часть маршурта, которая будет определять какие http запросы будут обрабатываться декорируемым методом. Может быть как одна строка, так и массив строк. Подробнее можно узанть в документации по "Werkzeug The Python WSGI Utility Library" (http://werkzeug.pocoo.org/docs/routing/ ).
      • type -- тип запроса, может быть 'http' или 'json'.
      • auth --

        Тип метода авторизации. может иметь следующие значения:

        • user: Пользователь должен быть авторизован и текущий запрос будет выполняться с правами данного пользователя.
        • public: Пользователь не обязан быть авторизован. Если он не авторизован, то текущий запрос будет выполнен от имени Публичного пользователя.
        • none: Метод всегда активен, даже если нет базы данных. В основном использоуется фреймворком и модулями авторизации. Код запроса не будет иметь никаких средств для доступа ни к базе данных ни какой либо информации указывающей на текущую базу данны и текущего пользователя.
      • methods -- Последовательность http методов, к которым пренадлежит данный маршрут. Если не указано иное, то все методы разрешены.
      • cors -- значание Access-Control-Allow-Origin cors дерективы.

      Запросы

      Объект запрос автоматически отправляется на openerp.http.request в самом начале запроса

      class openerp.http.WebRequest(httprequest)[source]

      Родительский класс для всех Odoo Web типов запросов, в основном имеет дело с инициализацие и настройкой объекта запроса (диспетчерезация как таковая выполняется подклассами)

      Парметры
      httprequest (werkzeug.wrappers.BaseRequest) -- упакованный объект werkzeug Request
      httprequest

      исходный werkzeug.wrappers.Request объект предоставляется на запрос

      params

      Mapping отображение параметров запроса не так полезно в отличии от того как они предоставляются непосредственно методу обработчика в качестве ключевых слов

      env

      Environment окружение связано с текущим запросом. Вызывает RuntimeError если текущий запрос не связан с базой данных.

      context

      Mapping отображение контекстных значений для текущего запроса

      session

      a OpenERPSession удержание данных HTTP сессии для текущй http сессии

      cr

      Cursor Курсор инициализации для текущего вызова метода.

      Доступ курсора, когда текущий запрос использует none параметр авторизиации вызовет исключение.

      debug

      Указывает, является ли текущий запрос находится в режиме "отладки"

      session_id

      непрозрачный идентификатор OpenERPSession экземпляра текущего запроса

      Устарел начиная с версии 8.0: Испльзуйте sid атрибут для session

      registry

      Запись в базе данных связанная с данным запросом. Может быть None если текущйи запрос использует none параметр авторизации.

      Устарел начиная с версии 8.0: используйте env

      db

      База данных связанная с данным запросом. Может быть None, если текущий запрос использует none параметр авторизации.

      httpsession

      данны HTTP сессии

      Устарел начиная с версии 8.0: Используйте session.

      class openerp.http.HttpRequest(*args)[source]

      Обработчик типа запроса http.

      совпадающие параметры маршрутизации, запрашивают строковые параметры, параметры формыform и файлы, передаются методу обработчика в качестве ключевых аргументов.

      В случае конфликта имен, параметр маршрутизации будет иметь приоритет.

      Результат обработчика метода может быть:

      • не существующие значение, в этом случае HTTP ответ будет HTTP 204 (Нет контента)
      • a werkzeug Response объект, который возвращется как есть (as-is)
      • a str или unicode, будет завернут в werkzeug Response объект и интерпретирован как HTML
      make_response(data, headers=None, cookies=None)[source]

      Помощь по не-HTML ответам, или HTML ответам со специфическими заголовками или куками.

      В то время как обработчики могут просто вернуть HTML разметку страницы, которую они хотят отправить в виде строки, если были возвращены не-HTML данные то им нужно создать полноценный объект ответа, или возвращенные данные будут не верно интерпретированы на строрне клиента.

      Параметры
      • data (basestring) -- тело ответа
      • headers ([(name, value)]) -- HTTP заголовки, отправляемые в ответе
      • cookies (collections.Mapping) -- куки для установки на клиенте
      not_found(description=None)[source]

      Ярлык для HTTP 404 (Страница не найдена) ответа

      render(template, qcontext=None, lazy=True, **kw)[source]

      Ленивый рендер QWeb шаблона.

      Фактический рендер предоставленного шаблона будет проиходить в самом конце диспетчерезации. В тоже время, шабло и/или qcontext могут быть изменены или даже заменены статическим ответом.

      Параметры
      • template (basestring) -- шаблон для рендера
      • qcontext (dict) -- контекст используемый при рендере
      • lazy (bool) -- следует ли отложить рендеринг данного шаблона до последнего момента
      • kw -- направление werkzeug Response объекту
      class openerp.http.JsonRequest(*args)[source]

      Запрос обработчика JSON-RPC 2 поверх HTTP

      • method игнорируется
      • params должен быть JSON объект (не массив) и передается в качестве ключевых аргументов в методе обработчика
      • результат метода обработчика возвращается как JSON-RPC result и обернут в JSON-RPC Response

      Успешный запрос:

      --> {"jsonrpc": "2.0",
     "method": "call",
     "params": {"context": {},
                "arg1": "val1" },
     "id": null}

<-- {"jsonrpc": "2.0",
     "result": { "res1": "val1" },
     "id": null}

      Запрос вызывающий ошибку:

      --> {"jsonrpc": "2.0",
     "method": "call",
     "params": {"context": {},
                "arg1": "val1" },
     "id": null}

<-- {"jsonrpc": "2.0",
     "error": {"code": 1,
               "message": "End user error message.",
               "data": {"code": "codestring",
                        "debug": "traceback" } },
     "id": null}

      Контороллеры

      Контроллеры необходимы для обеспечения расширяемости, так же как и модель данных, модель данных, но не может использовать то же механизм пред-запросов (база данных с загруженными модулями) может быть не доступен в текущий момен (например: не создана база данных, или база данных не выбрана).

      Таким образом, контроллеры обеспечивают свой собственный механизм расширения, незавизимо от моделей дыннах:

      Контроллеры создаются наследованием из

      class openerp.http.Controller[source]

      и определением методов, декорируемых route():

      class MyController(openerp.http.Controller):
    @route('/some_url', auth='public')
    def handler(self):
        return stuff()

      Для перегрузки контроллера, наследуйте его класс и перегрузиет соотвествующий метод, вновь используя их при необходимости

      class Extension(MyController):
    @route()
    def handler(self):
        do_before()
        return super(Extension, self).handler()
      • декорирование с помощью route() необходимо для того, чтобы сохранить метод (и маршрут) видимым: если метод переопределен без декорирования, он будет "неопубликован"

      • декораторы всех методов объеденены, если перегрузка декораторов метода не имеет аргумента, то все предыдущие будут сохранены, любой предоставленный аргумент будет перегружать ранее определенный:

        class Restrict(MyController):
    @route(auth='user')
    def handler(self):
        return super(Restrict, self).handler()

        изменит /some_url механизм авторизации с public на user (требующий подтверждения данных)
